# EfficientNet

> The paper introduced compound scaling for CNN depth, width, and input resolution.

## Metadata

| Field | Value |
| --- | --- |
| Paper | EfficientNet: Rethinking Model Scaling for Convolutional Neural Networks |
| Authors | Mingxing Tan, Quoc V. Le |
| Year | 2019 |
| Venue | ICML 2019 |
| arXiv | [1905.11946](https://arxiv.org/abs/1905.11946) |
| PMLR | [v97/tan19a](https://proceedings.mlr.press/v97/tan19a.html) |
| Status | verified |

## Question

CNNs can be made larger in several ways:

- deeper: more layers;
- wider: more channels;
- higher resolution: larger input images.

Earlier CNN papers often changed these dimensions manually. EfficientNet asks whether depth, width, and resolution should be scaled together under a principled compute budget.

The architecture question is:

$$
\text{Given a strong baseline CNN, how should model capacity be scaled as resources increase?}
$$

This is a scaling paper as much as an architecture paper.

## Main Claim

EfficientNet proposes compound scaling: jointly scale network depth, width, and input resolution using a single compound coefficient.

The scaling rule is:

$$
d = \alpha^\phi
$$

$$
w = \beta^\phi
$$

$$
r = \gamma^\phi
$$

subject to an approximate compute constraint:

$$
\alpha\cdot\beta^2\cdot\gamma^2 \approx 2
$$

with:

$$
\alpha\ge 1,\qquad \beta\ge 1,\qquad \gamma\ge 1.
$$

Here:

| Symbol | Meaning |
| --- | --- |
| $d$ | depth multiplier |
| $w$ | width multiplier |
| $r$ | input resolution multiplier |
| $\phi$ | compound scaling coefficient |
| $\alpha,\beta,\gamma$ | fixed scaling constants found by search |

The durable claim is:

$$
\text{balanced depth-width-resolution scaling}
\Rightarrow
\text{better accuracy/efficiency tradeoff than one-axis scaling}.
$$

## Architecture Contract

| Item | Contract |
| --- | --- |
| Input | image at scaled resolution |
| Output | image class probability |
| Baseline | EfficientNet-B0 from neural architecture search |
| Scaling dimensions | depth, width, resolution |
| Scaling family | B0 through larger variants |
| Core block | MBConv-style efficient convolution block |
| Claim type | accuracy/efficiency scaling under compute budget |
| Main caveat | FLOPs, parameters, and real latency are not identical |

EfficientNet is a two-part contribution:

1. find a strong baseline network;
2. scale it with a compound rule.

Do not collapse these into one claim.

## Why Compound Scaling?

Increasing only depth can increase representational capacity but may cause optimization difficulty and diminishing returns:

$$
\text{depth only}
\Rightarrow
\text{longer path and more layers}.
$$

Increasing only width can add channels but may not expand receptive-field hierarchy:

$$
\text{width only}
\Rightarrow
\text{more features per layer}.
$$

Increasing only resolution gives the network more spatial detail but also requires enough depth and width to process it:

$$
\text{resolution only}
\Rightarrow
\text{more pixels, higher compute}.
$$

EfficientNet argues these axes are coupled. If input resolution increases, the network should usually also have enough depth and width to use the extra detail.

## FLOPs Scaling Intuition

For a convolutional layer, a rough compute estimate is:

$$
\operatorname{FLOPs}
\propto
H W C_{\text{in}} C_{\text{out}} K^2.
$$

If input resolution scales by $r$, spatial area scales roughly as:

$$
H W \to r^2 HW.
$$

If width scales by $w$, channel product scales roughly as:

$$
C_{\text{in}}C_{\text{out}}\to w^2C_{\text{in}}C_{\text{out}}.
$$

If depth scales by $d$, number of repeated layers scales as:

$$
L\to dL.
$$

So total compute scales approximately as:

$$
\operatorname{FLOPs}
\propto
d\cdot w^2\cdot r^2.
$$

This explains the EfficientNet constraint:

$$
\alpha\cdot\beta^2\cdot\gamma^2\approx 2.
$$

Each unit increase in $\phi$ roughly doubles compute.

## Compound Coefficient

The model family is generated by increasing $\phi$:

$$
(d,w,r)
=
(\alpha^\phi,\beta^\phi,\gamma^\phi).
$$

For $\phi=0$:

$$
d=w=r=1
$$

which corresponds to the baseline scale.

For larger $\phi$, the model gets deeper, wider, and uses higher input resolution together.

The core design move is that scaling is not arbitrary per model. Once $\alpha,\beta,\gamma$ are chosen, the family follows a structured rule.

## Baseline And Scaling Are Separate

EfficientNet-B0 is found with neural architecture search. The larger EfficientNet variants are produced by compound scaling.

This creates two claims:

| Claim | Evidence Needed |
| --- | --- |
| B0 is a strong baseline | compare against similarly sized CNNs |
| compound scaling is better | compare against depth-only, width-only, resolution-only scaling |

When reading the paper, keep these separate. A strong baseline can make scaling look good, and a good scaling rule can make a baseline family more useful.

## MBConv-Style Building Blocks

EfficientNet uses mobile inverted bottleneck-style blocks. The common pattern is:

$$
1\times1\text{ expand}
\rightarrow
\text{depthwise }K\times K
\rightarrow
1\times1\text{ project}.
$$

Depthwise convolution applies spatial convolution per channel:

$$
Y_{u,v,c}
=
\sum_{\Delta u,\Delta v}
W_{\Delta u,\Delta v,c}
X_{u+\Delta u,v+\Delta v,c}.
$$

Pointwise $1\times1$ convolutions mix channels:

$$
Y_{u,v,k}
=
\sum_c W_{c,k}X_{u,v,c}.
$$

This separates spatial filtering from channel mixing, reducing compute relative to full convolution.

## Block View

| Component | Role | Architecture Implication |
| --- | --- | --- |
| EfficientNet-B0 | searched baseline | scaling starts from a strong seed |
| compound coefficient $\phi$ | resource scale index | controls family size |
| depth multiplier $d$ | repeats/stacks more layers | increases hierarchy and path length |
| width multiplier $w$ | increases channels | increases feature capacity |
| resolution multiplier $r$ | increases input size | preserves more spatial detail |
| MBConv blocks | efficient local processing | strong mobile-CNN prior |
| squeeze/excitation-style channel attention | channel recalibration | extra capacity in blocks |

The paper's architectural importance is in the family-level scaling rule, not only the block definition.

## Relation To Inception

[[papers/architectures/inception|Inception]] is manually compute-aware: it uses branches and $1\times1$ projections to control cost.

EfficientNet is scaling-aware:

$$
\text{given a baseline, scale }(d,w,r)\text{ together}.
$$

| Dimension | Inception | EfficientNet |
| --- | --- | --- |
| Main design question | how should a module allocate compute? | how should a model family scale? |
| Mechanism | multi-branch modules and bottlenecks | compound scaling |
| Search role | manual design | NAS baseline plus scaling |
| Output | GoogLeNet/Inception family | EfficientNet B-family |

Both are compute-aware, but at different levels.

## Relation To ResNet And DenseNet

[[papers/architectures/deep-residual-learning|ResNet]] and [[papers/architectures/densenet|DenseNet]] focus on connectivity patterns that make deeper CNNs trainable and reusable.

EfficientNet asks how to scale a CNN family once the base architecture is set.

| Paper | Main Question |
| --- | --- |
| ResNet | how can very deep networks optimize? |
| DenseNet | how can features be reused through dense connectivity? |
| EfficientNet | how should depth, width, and resolution scale together? |

This makes EfficientNet a family-scaling paper rather than just another backbone block paper.

## Relation To Vision Transformers

[[papers/architectures/vision-transformer|Vision Transformers]] scale differently. For self-attention, token count strongly affects compute:

$$
\operatorname{AttentionCost}
\sim
O(N^2d).
$$

For CNNs, resolution affects convolutional compute roughly through spatial area:

$$
O(HW C^2 K^2).
$$

EfficientNet is useful because it makes the CNN scaling axes explicit. When comparing CNNs and ViTs, ask whether the scaling rule controls:

- resolution;
- parameter count;
- FLOPs;
- training recipe;
- pretraining data;
- actual latency.

## FLOPs vs Latency

EfficientNet reports strong FLOPs/parameter tradeoffs, but real deployment depends on hardware and kernels.

FLOPs can miss:

- memory bandwidth;
- kernel fusion;
- depthwise convolution efficiency;
- batch size;
- accelerator support;
- input pipeline overhead;
- quantization behavior.

So the safe reading is:

$$
\text{good FLOPs/accuracy}
\not\Rightarrow
\text{best latency on every device}.
$$

## Training Recipe Boundary

EfficientNet's results depend on more than architecture:

- input resolution;
- augmentation;
- regularization;
- optimizer and schedule;
- model scaling;
- baseline search;
- transfer fine-tuning setup.

Architecture comparisons should control these. If a new model changes both architecture and recipe, attribution is weak.

## Implementation Notes

When using or comparing EfficientNet-style models, check:

| Detail | Why It Matters |
| --- | --- |
| variant B0-B7 or later | determines $\phi$ scale |
| input resolution | central part of scaling |
| width/depth multipliers | architecture capacity |
| MBConv expansion ratio | compute and channel bottleneck |
| squeeze-excitation settings | block capacity and latency |
| dropout/drop-connect | regularization changes with scale |
| preprocessing | affects reported ImageNet accuracy |
| FLOPs vs latency | hardware-specific tradeoff |
| pretrained weights | recipe and dataset matter |

EfficientNet is easy to misuse if only the model name is reported without resolution and recipe.

## Common Misreadings

### "EfficientNet is only a better CNN block"

The key contribution is compound scaling of a searched baseline family, not just a block.

### "FLOPs prove efficiency"

FLOPs are useful but incomplete. Hardware latency and memory behavior can differ.

### "The scaling rule is universal"

The rule was tested for a specific CNN family and setup. Other families may need different scaling constants.

### "Bigger resolution is always better"

Resolution helps only if the model has enough capacity and the task benefits from the extra spatial detail.

## What To Check In Later Scaling Papers

- What is the baseline architecture?
- Are depth, width, resolution, and data all scaled?
- Is compute reported as FLOPs, wall time, memory, or energy?
- Are training recipes held fixed?
- Is the result single-model or ensemble?
- Does scaling transfer to dense prediction or only classification?
- Are latency results measured on the target hardware?

## Why It Still Matters

EfficientNet made CNN scaling explicit:

$$
\text{scale depth, width, and resolution together}.
$$

For this wiki, it completes a useful classic CNN path:

- [[papers/architectures/alexnet|AlexNet]]: large CNNs work.
- [[papers/architectures/vgg|VGG]]: small filters and depth work.
- [[papers/architectures/inception|Inception]]: modules can allocate compute across branches.
- [[papers/architectures/deep-residual-learning|ResNet]]: residual paths make depth trainable.
- [[papers/architectures/densenet|DenseNet]]: dense connectivity reuses features.
- EfficientNet: a CNN family should scale depth, width, and resolution together.

## Limitations

- Baseline architecture is NAS-derived, so search space matters.
- FLOPs and parameters do not fully determine real latency.
- Scaling constants are not guaranteed universal.
- Input resolution and training recipe are central to reported gains.
- Later architectures and recipes changed the efficient-vision landscape.
- The paper is classification-centric compared with broad modern vision evaluation.
